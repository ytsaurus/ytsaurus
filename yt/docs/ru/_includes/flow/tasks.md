## Инкрементальное обновление состояний различных объектов

Пример подобных сервисов &mdash; BigB, CaeSaR, HitMaster.

На входе &mdash; множество (десятки) различных логов, в каждой строке содержится ключ `ProfileID`.

Цель: поддерживать актуальное состояние `ProfileID` => `Profile`, описав функции применения строк на профиль. При этом необходимо делать батчевые запросы во внешние системы (синхронно и асинхронно), включая запросы с передачей самих профилей.

Поток входных логов &mdash; 1–3 ГБ/с, размер состояния &mdash; ~ 100 ТБ.

## Подготовка логов для ML

{% if audience == "internal" %}Пример сервиса &mdash; Colibri.{% endif %}

Цель: построить лог вида `(is_click, factors)` в результате джойна логов с факторами с рекламным event'ом по `JoinID`.

- Учитываются только клики/показы в течение часа после хита (джойн на временном окне).
- Клики нужны все (это несколько процентов от всех событий), показы нужно сэмплировать (до примерно сравнимого с кликами количества).
- Входной поток факторов составляет 15–20 ГБ/с в сжатом виде.

## Список роботов

Прямой аналог АА2 задачи на примере задачи BigB.

Есть поток запросов в сервис (2 млн [RPS](*rps), около 10 млн [KPS](*kps)).

Цель: в реальном времени показывать топ-100 запрашиваемых ключей.

## RT ABT

Есть поток событий (~ 1 млн RPS). Из каждого события извлекается до 10 инкрементальных метрик, до 100 testid (экспериментов) и до 10 разрезов (slice).

Цель: на выходе получать динтаблицу [`TestID`, `TimeWindow(5min)`, `Slice`, `Metric`, `Value`] для эффективных запросов значений метрик.

## RT Автобюджет

{% if audience == "internal" %}Как компонент [CaeSaR](https://nda.ya.ru/t/F2x4cUph7gKZHn).{% endif %}

Автобюджет управляет кампанией так, чтобы её показатели (потраченный бюджет, стоимость конверсий и т. п.) были в заданных рекламодателем показателях. Например, если у кампании слишком мало показов, то Автобюджет увеличивает ставку. Если показов/кликов/конверсий избыточно или они неоправданно дороги &mdash; снижает.

Для этого ему критично понимать полноту обработанных логов: как минимум отличать отсутствие событий по кампании от отсутствия логов.

Кроме того, для эффективного предсказания данные из разных источников должны быть консистентны между собой, то есть содержать информацию на один момент времени. Если в профиле будет число кликов до 13:00, а число конверсий &mdash; до 12:30, то это хуже, чем если бы в профиле лежала информация ровно на 12:30.

## RT Антифрод

Система обрабатывает некоторый лог событий, на выходе размечает его флагами фрода.

Особенности:

- Возможность описывать правила YQL-like.
- Подсчёт статистик по десятку ключей.
- Для принятия решения по одному событию необходимо посмотреть «в будущее» хотя бы на минуту (чтобы банить в том числе и первый клик из фрод-серии).
