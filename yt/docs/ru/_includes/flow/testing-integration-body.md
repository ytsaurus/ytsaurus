## Общие принципы {#principles}

Общие принципы написания интеграционных тестов на пайплайны:
* Тестируем конечные сценарии. То есть:
  * Записываем входные/исходные данные в локальный YT/LB.
  * В спеке пайплайна отмечаем источники как `finite=%true`.
  * Запускаем пайплайн.
  * Ждём, пока пайплайн завершится.
  * Проверяем выходные данные.
* Готовим окружение тем же кодом, что и в проде:
  * Тестируем тот же бинарь пайплайна что будет работать в проде.
  * Генерируем спеки пайплайна тем же кодом, что будет генерировать их для прода.
{% if audience == "internal" %}  * Используем продовый бинарь [YtSync]({{yt-sync-docs}}/), в котором есть отдельный стейдж для тестов.{% endif %}
  * Если входные/выходные данные нетривиально сериализуются/парсятся, то делаем эту работу общим с продом кодом.
* Используем [общий тестовый фреймворк]({{source-root}}/yt/yt/flow/library/python/integration_test_base) (есть `README.md`).
* Делаем failover тест.
  * Многие ошибки вскрываются на выпадении воркеров и переподхвате их работы другими воркерами. Поэтому делаем тест с `problems=True` и с более чем 1 воркером.
* Пишем стабильные тесты.
  * Помним, что в CI любая часть теста может работать неожиданно долго.
    * Идеальное время работы одного теста &mdash; 20 секунд при любом типе сборки.
    * Для санитайзерных сборок уменьшаем количество входных данных.
    * Все локальные таймауты выставляем с кратным запасом.
  * Логика выполнения теста не должна значимо зависеть от текущего времени. Например, тест не должен падать, если запустился до полуночи, а завершился &mdash; после.

## YtSync в тестах {#yt-sync}

Запускать [YtSync]({{yt-sync-docs}}/) в пользовательских тестах стоит через метод тестового фреймворка `run_yt_sync_ensure` как в [примере]({{source-root}}/yt/yt/flow/examples/cpp/wait_click_join/test/test_wait_click_join.py#L142). Этот метод запускает исполняемый файл YtSync с параметрами повышающими эффективность работы YtSync и с нужными переменными окружения. Запуск YtSync через `yatest.common.execute` тоже допускается, но без необходимости лучше не использовать этот вариант.

Если пайплайн существует исключительно для тестов, чтобы тестировать какой-то функционал YT Flow, то применяется _встроенный_ YtSync как в [примере]({{source-root}}/yt/yt/flow/tests/flow_execute/test_flow_execute.py#L33).

## Отладка тестов {#debug}

### Логи {#debug-logs}

После завершения работы теста его логи можно найти в `test-results/py3test/testing_out_stuff` в директории теста. Основные важные:
* `run.log` &mdash; логи python-теста.
* `<test_class_name>/<test_name>/Controller_<number>...` &mdash; логи контроллеров (`.err` &mdash; это stderr процесса, `.log` &mdash; обычные логи, которые пишутся через `YT_LOG_...`).
* `<test_class_name>/<test_name>/Worker_<number>...` &mdash; логи воркеров.
* `<test_class_name>/<test_name>/Runner...` &mdash; логи раннера.

Если что-то не работает, то имеет смысл поискать ошибки во всех этих логах.{% if audience == "internal" %} В целом, принцип изучения логов такой же как и на проде, описан на [странице диагностики проблем](../../flow/release/problems.md).{% endif %}

Логи можно посмотреть ещё до завершения теста, для этого нужно найти временную директорию, в которой тест работает. Самый простой способ — запускать тест с флагом `--keep-temps`: `ya make --keep-temps -ttt <target>`. В этом случае `ya make` не удалит временную директорию по окончании теста и напечатает ссылку на неё в выводе.

Также можно воспользоваться такими bash-алиасами:

```bash
alias curtestdir="ps -f -u $USER | python3 -c \"import sys, re; drs = set(e for e in re.findall(r'[\s=](/\S*testing' + r'_out_stuff)\b', sys.stdin.read())); print('' if len(drs) == 1 else 'Select first from ' + repr(drs), file=sys.stderr); print(list(drs)[0])\""
alias cdcurtestdir='cd $(curtestdir)'

# Достать из логов ссылку на UI локального {{product-name}}.
alias curlocalyt='cat $(curtestdir)/stderr 2>/dev/null | grep YT'
```

### UI локального {{product-name}} {#debug-ui-yt}

Нужны сетевые доступы от сети `_YTFRONT_PROD_NETS_` до машинки, на которой запускается тест, на tcp-порты 81-65535. Это даст возможность использовать продакшн и beta UI. В случае, если хочется использовать тестовый UI, то нужны ещё сетевые доступы  от сети `_DATAUI_INFRASTRUCTURE_NETS_` на те же порты.

В этом случае можно открывать обычный UI {{product-name}} для взаимодействия с пайплайном, запущенным на локальном {{product-name}}. Адрес UI локального кластера можно извлечь из логов, смотрите bash-алиасы в предыдущем пункте.

Для дебага с помощью UI нужно иметь большие таймауты в тестах (или вообще отключить их).
