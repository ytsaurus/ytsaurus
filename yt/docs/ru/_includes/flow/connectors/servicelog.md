# Сервис-лог в {{product-name}} Flow

Код коннектора находится [здесь]({{source-root}}/yt/yt/flow/library/cpp/connectors/servicelog).

Сервис-лог предназначен для регулярного обхода сортированных таблиц. В отличие от остальных коннекторов, он «бесконечный» по определению: после того как таблица «дочитана», он начинает читать её заново.

Также существует возможность параллельно обходить несколько таблиц, приджойнивая их друг к другу по ключевым колонкам.

{% note warning "Важно" %}

В настоящий момент требуется, чтобы первой колонкой в ключе было значение типа `ui64`, равномерно распределённое между `0` и `(1 << 64) - 1` (обычно в качестве такого значения можно взять хеш от id строки).

{% endnote %}

{% note warning "Важно" %}

Сорс можно запустить в финитном режиме установкой параметра `finite` в `true`. В этом режиме таблица будет прочитана ровно один раз, или (если ранее чтение уже началось) каждая [партиция](../../../flow/concepts/glossary.md#partition) дочитает 1 раз свою часть таблицы.

Этот режим предназначен главным образом для упрощения написания тестов.

{% endnote %}

## Как работать с реплицированными таблицами

Сервис-лог поддерживает реплицированные таблицы.

Для походов в них напрямую необходимо указать все кластера-реплики в атрибуте `clusters` таблицы.

Для походов через RPC мета-кластера достаточно указать его в атрибуте `cluster`.

{% note warning "Важно" %}

Походы через RPC мета-кластера могут осуществляться только с `FetchType = EFetchType::SelectRows`

{% endnote %}

## Фулстейты {#fullstates}

В сервис-логе можно указывать несколько фетчеров, это можно использовать для сценария "фулстейта" ("fullstate"). Фулстейт &mdash; это внешняя по отношению к [пайплайну](../../../flow/concepts/glossary.md#pipeline) таблица с данными по тем же [ключам](../../../flow/concepts/glossary.md#key), по которым есть [стейт](../../../flow/concepts/glossary.md#state) в пайплайне. Эта таблица периодически "заливается" в пайплайн. Это можно делать эффективным образом: при генерации сервис-лога сразу джойнить по ключам таблицу стейтов пайплайна и таблицу фулстейта. Таким образом будут генерироваться события по ключам, которые есть хотя бы в одном из фетчеров. За счёт этого можно достичь двух полезных свойств:
* За один проджойненный проход сервис-лога каждый стейт трогается один раз. Вместо двух раз &mdash; один на основное событие сервис-лога и один на заливку строчки фулстейта.
* В пайплайне можно реагировать на события вида: ключ есть стейте пайплайна, но его нет во внешнем фулстейте.

Отличить случай "все неключевые колонки null" от случая "нет вхождения по ключу" можно по служебной колонке `ispresent`.

## Как писать тесты

Распространенная ошибка &mdash; писать тесты к сервис-логу в финитном (`finite = true`) режиме, ожидая, что он увидит данные, возникшие вследствие обработки сообщений из других сорсов. При таком подходе невозможно гарантировать, что при обходе сервис-лога он увидит ожидаемые строки: может случиться, что он "просканирует" некоторый диапазон ключей до того, как в нем появятся строки. Вместо этого есть несколько подходов, как писать тесты корректно:

1. Залить строки, которые хочется, чтобы сервис-лог увидел, руками до начала теста (при таком подходе важно следить, чтобы результат теста не зависел от того, увидит он или нет остальные строки).
2. Сделать 2 запуска: в первом залить данные по другим сорсам, во втором запустить только сорс сервис-лога.
3. Сделать тест нефинитным (`finite = false`).

{% note warning "Важно" %}

При использовании `FetchType = EFetchType::TableReader` необходимо указывать на таблице `enable_dynamic_store_read = true`, в противном случае сервис-лог может не сразу видеть наиболее свежую версию данных (обычно это некритично для прода, но ломает тесты).

{% endnote %}

В статической спеке сервис-лога задается список фетчеров таблиц, которые необходимо обходить, джойня их по ключевым колонкам. Чаще всего фетчер ровно один и сорс просто обходит одну таблицу.

{% include [NYT_NFlow_TTableJoinerSpec](../../../flow/generated_docs/NYT_NFlow_TTableJoinerSpec.md) %}

В настройках фетчера указывается, какую таблицу обходить, какие колонки брать, с каким префиксом эти колонки складывать в результат джойна (если джойн имеет место).

##### Пример минимальной спеки

```
"table_joiner" = {
    "fetchers" = [
        {
            "table_path" = <путь к таблице>;
        };
    ];
};
```
