# Базовые правила выкатки Pipeline в {{product-name}} Flow

Этот раздел описывает общие правила выкатки релизов и изменения конфигурации в {{product-name}}, не зависящие от способа запуска.

## Как запустить {#launch-flow}

Контроллеры и воркеры можно поднять следующими способами:

* [Запуск в Vanilla-операции](../../../flow/release/launch-vanilla.md) — контроллеры и воркеры поднимаются внутри одной {{product-name}} vanilla-операции, отдельный долгоживущий деплой не нужен. Самый простой способ.{% if audience == "internal" %}
* [Запуск через Infractl](../../../flow/release/launch-infractl.md) — долгоживущий деплой контроллеров и воркеров в YP. Рекомендуется для продакшена.{% endif %}

## Изменение конфигурации таблиц в {{product-name}} {#yt-sync}
При первом запуске пайплайна или при его обновлении часто требуется изменять конфигурацию объектов в {{product-name}} &mdash; например, добавлять новые таблицы, удалять старые или изменять схему уже существующих таблиц. Как это сделать, разобрано на [примере wait_click_join](../../../flow/cpp/examples/wait_click_join.md#yt-sync).{% if audience == "internal" %} Удобнее всего эти операции делать через [YtSync]({{yt-sync-docs}}/), подробнее про который можно почитать в его документации.{% endif %}

Обновление конфигурации объектов в {{product-name}}, особенно выполнение [миграций](../../../flow/concepts/glossary.md#migration), &mdash; потенциально опасная операция, поэтому её запуск находится в зоне ответственности пользователя Flow. Правила, которые необходимо при этом соблюдать, описаны ниже.

## Общие правила выкатки релизов пайплайнов и изменения конфигурации на {{product-name}} {#release-and-configure-basic-rules}

Вне зависимости от способа управления объектами в {{product-name}}, при выкатке [релиза пайплайна](../../../flow/concepts/glossary.md#release-pipeline) соблюдайте следующие правила:

- Иметь в своем релизном пайплайне (в процедуре выкатки нового релиза пайплайна) проверку диффа{% if audience == "internal" %} ([сценарий dump-diff в YtSync]({{yt-sync-docs}}/getting_started#dostupnye-scenarii#dump-diff-scenario)){% endif %} между актуальным состоянием объектов (их существование, схема, атрибуты) в {{product-name}} и состоянием, необходимым для работы новой версии пайплайна{% if audience == "internal" %} (это состояние описывается в `YtSync`){% endif %}.

    Так как в новом релизе могут, например, использоваться новые таблицы, то, если они еще не созданы, пайплайн Flow не сможет запуститься (будет падать с ошибками), что приведет к нежелательному даунтайму. Обязательная проверка диффа перед обновлением статической спеки и исполняемых файлов пайплайна может сильно уменьшить вероятность подобных ошибок.

    {% note warning "Внимание" %}

    При выкатке нового релиза могут измениться схемы внутренних таблиц Flow, в этом случае обязательно не забыть выполнить [миграцию](../../../flow/concepts/glossary.md#migration) при полной остановке пайплайна (обязательно не забыть и обязательно при полной остановке).

    {% endnote %}

- Все [миграции](../../../flow/concepts/glossary.md#migration) системы осуществлять при полной остановке пайплайна.

    Правило сильно повышает безопасность миграций, особенно привязанных к релизу. И упрощает разбор инцидента, если что-то пойдет не так.

    В небольшом количестве случаев это правило можно нарушать, чтобы уменьшить время даунтайма:

    - Новые пользовательские таблицы можно создавать до обновления пайплайна. Иногда и новые колонки тоже, но только если есть точное понимание, что в результате ничего не сломается.
    - Старые пользовательские таблицы можно удалять после обновления пайплайна, проверив, что их никто не использует. Старые неиспользуемые колонки тоже можно так удалять, но только если есть точное понимание, что в результате ничего не сломается.

- Не выкатывать релизы пайплайна и не проводить операции над таблицами во время работ на кластерах {{product-name}}.

- Поддерживать по возможности одинаковые настройки таблиц на разных [окружениях](../../../flow/concepts/glossary.md#environment) (продакшн/препродакшн/тестинг), по крайней мере не иметь ненужных различий.

## Настройка аутентификации {#authentication}

{% if audience == "internal" %}Есть два способа аутентификации для взаимодействия с {{product-name}}: TVM и OAUTH.

TVM &mdash; более универсальный способ: его достаточно один раз настроить, а дальше можно просто выдавать права, и он будет использоваться для аутентификации в {{product-name}}, Logbroker, Monitoring (tracing) и т. д.{% else %}Аутентификация для взаимодействия с {{product-name}} выполняется по OAUTH-токену.{% endif %}

{% note warning "Внимание" %}

Необходимо, чтобы пользователь (робот){% if audience == "internal" %} или TVM-приложение{% endif %} имел хотя бы одну роль на кластере YT (как минимум с доступами до директории пайплайна). В противном случае возникнет ошибка отсутствия пользователя{% if audience == "internal" %}/TVM-приложения{% endif %}.

{% endnote %}


{% if audience == "internal" %}

### TVM {#authentication-tvm}

1. Заведите TVM-приложение для вашего сервиса ([инструкция](https://docs.yandex-team.ru/tvm/pages/getting_started)), если его ещё нет.
2. Запускайте контроллеры и воркеры с переменными окружения `TVM_ID` и `TVM_SECRET`, в которые положите соответствующие значения.

{% endif %}

### OAUTH {#authentication-oauth}

1. Заведите робота.
2. Выпустите для робота токен {{product-name}}.
3. Запускайте контроллеры и воркеры с переменными окружения `YT_USER` и `YT_TOKEN`, в которые положите логин робота и его {{product-name}}-токен.
