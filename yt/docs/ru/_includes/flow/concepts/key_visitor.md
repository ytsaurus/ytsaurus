# Key Visitor Streams в {{product-name}} Flow

## Зачем нужны key-visitor-стримы {#why-key-visitors}

Stateful-компьютейшены накапливают [внутренний стейт](../../../flow/concepts/glossary.md#state) на ключ, обрабатывая входящие сообщения. Многие задачи требуют периодически *обойти* этот стейт — даже без входящего сообщения по конкретному ключу:

- **TTL / eviction**: пройти все ключи и удалить устаревшие записи.
- **Периодические агрегаты**: для каждого ключа эмитнуть текущее значение наружу раз в сутки.
- **Принудительная переоценка**: триггернуть пересчёт по всем ключам в фоне.

Стандартные таймеры (см. [Таймеры](../../../flow/concepts/timers.md)) тут не подходят: их нужно регистрировать на каждый ключ заранее, а множество ключей может расти без явного события «новый ключ появился» (например, ключи приходят только через стейт-сторы, без message-стрима).

Key-visitor-стрим решает эту задачу: фоновая задача в воркере периодически *сканирует* весь стейт партиции и для каждого ключа эмитит сообщение типа `TVisit` в специальный внутренний стрим. Компьютейшен подписывается на этот стрим через `DoProcessVisit` / `process_visit` и принимает решение по своему стейту так же, как на обычное входящее сообщение.

## Как это работает {#how-it-works}

### Жизненный цикл прохода (Pass) {#pass-lifecycle}

1. **Бэкграунд-фил.** На каждой партиции работает фоновый цикл: он читает страницами стейт через `KeyStates::List`, отдаёт ключи дальше в внутренний буфер визитов. Скорость регулируется троттлером, сконфигурированным так, чтобы один полный обход (*pass*) занимал заданный `Period`.
2. **Эмит.** Готовые `TVisit` отдаются движком через `GetNextBatch`, попадают компьютейшену в `DoProcessVisit`.
3. **Покрытие.** После того как визиты для диапазона ключей отдали потребителю, диапазон помечается как *Committed* в `TKeyVisitorStore`. Покрытие персистится в системную таблицу `key_visitor_states`, поэтому restart воркера или ребаланс партиций не приводит к повторному обходу.
4. **Конец прохода.** Когда покрытие полное, фоновый цикл сразу вызывает `StartNewPass` — темп задаётся троттлером, так что следующий обход всё равно займёт `Period`. Ротация атомарна: одна Sync-транзакция стирает строки прошлого pass'а и сеет первый интервал нового; crash посередине откатывается, покрытие сохраняется.
5. **Финальный pass.** Когда все входные стримы Completed, следующий pass помечается *Final*. После его коммита visit-стрим становится `Empty` и новых pass'ов не запускает — гарантирован **минимум один** полный обход после завершения входов.

### Партиционирование {#partitioning}

Стейт партиционируется по uint64 хэшу первой колонки `group_by_schema` (обычно `farm_hash(key)`). Каждая партиция собственным `TKeyVisitor`-ом обходит свой хэш-диапазон.

Внутри партиции диапазон делится на статически заданное число *бакетов* (`bucket_count`). Бакеты обходятся round-robin: это сглаживает нагрузку и держит «незавершённое покрытие» равномерно распределённым по партиции, чтобы крах воркера не отбрасывал прогресс в одном узком месте.

## Гарантии {#guarantees}

| Свойство | Гарантия |
|---|---|
| Каждый ключ за период | Visit ровно один раз (без дублей и пропусков). |
| Перезапуск воркера | Покрытие сохраняется; ротация атомарна, crash сохраняет старый pass. |
| Ребаланс партиций | Новый воркер видит закоммиченное покрытие через `key_visitor_states`. |
| Завершение входов | Гарантированно произойдёт минимум один финальный pass; `Empty` объявляется только после него. |
| `Period` | Best-effort. Под нагрузкой троттлера или медленных чтений `KeyStates` достигнутый период растёт. См. ниже про наблюдаемость. |
| Порядок обхода | Внутри бакета — отсортированный по ключу; между бакетами — round-robin. Не event-time. |

## Какие компьютейшены поддерживают visit-стримы {#supported-computations}

| Computation | Поддержка visit-стримов |
|---|---|
| `TTransformComputation` | ✓ |
| `TSwiftMapComputation` | ✓ (только для работы со стейтом: эмит в output из `DoProcessVisit` запрещён, см. [Swift](../../../flow/concepts/swift.md#swift-map)) |
| `TSwiftOrderedSourceComputation` | ✗ |

## Конфигурация {#configuration}

Чтобы компьютейшен принимал visit-стрим, в его спеке (статической) необходимо заполнить поле `key_visitor_streams`:

```yson
"tester" = {
    "computation_class_name" = "...";
    "group_by_schema" = [...];
    "key_visitor_streams" = {
        "visit_iter" = {};
    };
    ...
};
```

### Требования к схеме {#schema-requirements}

- `group_by_schema` обязан начинаться с колонки типа `uint64` — она используется как хэш для разделения партиции на бакеты. Проверка выполняется на этапе submit'а спеки.

### Параметры статической спеки {#static-params}

- `names` — список имён internal-стейтов, по которым считать ключи. Если не задан или пустой — все ключи всех internal-стейтов компьютейшена.
- `external_names` — список имён [external-стейт-менеджеров](../../../flow/concepts/glossary.md#state) компьютейшена, по таблицам которых считать ключи. Позволяет визитору обходить external-стейт (в том числе у companion-компьютейшенов) и эвиктить устаревшие записи через `clear()` в обработчике визита. Если задан только `external_names` (без `names`) — обходится только перечисленный external-стейт, internal-стейты не сканируются. Если не заданы ни `names`, ни `external_names`, обходятся все internal-стейты, а external-стейт — нет.

### Параметры динамической спеки {#dynamic-params}

В `dynamic_spec.computations.<id>.key_visitor_streams.<name>`:

- `period` — целевое время одного полного прохода. Дефолт — 1 день.
- `scan_batch_size` — лимит на один `KeyStates::List` вызов (в строках, не ключах). Должен быть строго больше, чем максимальное число имён internal-стейтов на один ключ, иначе обход стопорится (см. [Диагностика](#diagnostics)).
- `buffer_row_limit` — максимальный размер внутреннего буфера готовых визитов между бэкграунд-филом и `GetNextBatch`.

### Компьютейшен только с visit-стримом {#key-visitor-only}

Key-visitor-стрим может быть **единственным** источником работы компьютейшена: `input_stream_ids` пуст, `source_streams` нет, работа приходит только из обхода external-стейта (`external_names`). Это паттерн ревизора: периодически переоценивать ключи external-таблицы без message-стрима. Такой компьютейшен партиционируется по диапазонам uint64-хэша так же, как input-driven (см. [Партиционирование](#partitioning)); число партиций задаётся `min_partition_count` / `max_partition_count` / `desired_partition_count` в динамической спеке.

Требование к `group_by_schema` (первая колонка `uint64`, см. [Требования к схеме](#schema-requirements)) для такого компьютейшена обязательно — по ней строятся диапазоны партиций.

{% note warning %}

**Одна external-таблица — один писатель.** Визитор-компьютейшен модифицирует external-стейт. Не модифицируйте одну и ту же external-таблицу из нескольких компьютейшенов: конкурентная запись из разных компьютейшенов не сериализуется между собой и приводит к гонкам и потере обновлений. На этапе валидации спеки проверяется владение YT-путями: если два компьютейшена объявляют запись (через свои external-стейт-менеджеры) в один и тот же `(cluster, path)`, submit спеки падает с ошибкой `claimed for writing`.

{% endnote %}

### `streams_dependency` {#deps}

По умолчанию visit-стрим **не входит** в `streams_dependency` компьютейшена: визиторы обычно нужны для внутреннего клинапа и не порождают output, а зависший визитор не должен блокировать завершение output-стримов. Сигнал «последний проход» доставляется визитору воркером локально (`SetUpstreamCompleted`) — это не требует ребра в графе.

Если же компьютейшен **эмитит сообщения в output из `DoProcessVisit`**, visit-стрим необходимо прописать как parent этого output-а в `streams_dependency` явно. Пример:

```yson
"streams_dependency" = {
    "visits" = ["keys"; "visit_iter"];
};
```

— здесь `visits` — output-стрим, `keys` — input, `visit_iter` — key-visitor стрим. У компьютейшена только с visit-стримом (без input) parent один — `"visits" = ["visit_iter"]`.

## Наблюдаемость {#observability}

### EventTimestamp визитов {#event-timestamp}

Каждый эмитированный `TVisit` несёт `EventTimestamp = ожидаемое время обхода этого ключа`. Формула:

```
scheduleLag = max(0, elapsed − ScannedFraction · Period)
EventTimestamp = now − scheduleLag
```

— где `elapsed` — время с начала текущего прохода (берётся как минимум `PassStartedAt` среди уже отсканированных интервалов; персистится в `key_visitor_states`, поэтому переживает restart и ребаланс).

Если обход идёт по графику — `scheduleLag = 0`, и `EventTimestamp ≈ SystemTimestamp`. Если отстаём (например, троттлер режет из-за медленного бэкенда) — `EventTimestamp` лежит в прошлом, и watermark visit-стрима отстаёт ровно на величину отставания. Это даёт прямой сигнал downstream-потребителю и в стандартных flow-метриках event-lag.

### Диагностика {#diagnostics}

- **`scan_batch_size` слишком мал.** Если у одного ключа имён internal-стейтов больше, чем `scan_batch_size`, обход не может продвинуться: лимит выпадает в середине ключа, все его строки сбрасываются, а с прогрессом не происходит. Компьютейшен выставляет ошибку `/key_visitor/<stream>/scan_batch_too_small` через `StatusProfiler` и периодически пишет warning в лог воркера (`Key visitor stalled (scan_batch_size too small)`). Решение — повысить `scan_batch_size` через `Reconfigure`.
- **Бэкенд состояния недоступен.** Если чтение стейта во время обхода падает, воркер не роняется: ошибка выставляется в `/key_visitor/<stream>/background_fill`, а чтение автоматически повторяется. Временная ошибка YT/RPC при повторе уходит сама; устойчивая (расхождение схемы стейта, отсутствующая таблица) остаётся видна в статусе и не роняет пайплайн.
