# YT-прагмы

YT‑прагмы — это пространство имён для прагм, которые настраивают специфичные для {{product-name}} параметры YQL‑запросов.

## Синтаксис

Названия YT‑прагм содержат префикс `yt`:

```yql
PRAGMA yt.<pragma_name> = '<value>';
```

{% note warning %}

Передавайте значения всех YT‑прагм как строки, в кавычках, вне зависимости от типа. Используйте одинарные `'...'` или двойные `"..."` кавычки.

Например: `PRAGMA yt.QueryCacheUseExpirationTimeout = 'true';`

{% endnote %}

## Область действия и особенности

По области действия YT‑прагмы можно разделить на статические и динамические.

**Статические прагмы**:

- инициализируются один раз на самой ранней стадии обработки запроса;
- действуют на все выражения текущего модуля, в котором они объявлены;
- если статическую прагму указать в запросе несколько раз, применится только её последнее заданное значение.

**Динамические прагмы**:

- инициализируются на стадии исполнения запроса, после его оптимизации и построения плана исполнения;
- действуют до следующей такой же прагмы или до конца запроса;
- только для динамической прагмы можно сбросить её значение в значение по умолчанию через присваивание `default`.

{% note info %}

Все прагмы, которые влияют на оптимизаторы запроса, являются статическими, так как значения динамических прагм на этой стадии ещё не вычислены.

{% endnote %}

## Поддержка per-cluster {#settings}

Некоторые прагмы поддерживают специальный режим работы — _per-cluster_. Этот режим позволяет  в одном запросе задать разные значения прагм для разных кластеров. Например, можно указать: «На кластере А выполнить запрос с настройками X, а на кластере В — с настройками Y». Это полезно в распределённых запросах, где разные кластеры требуют разных условий выполнения.

Режим _per-cluster_ доступен для всех динамических прагм и для некоторых статических — в документации ниже такие прагмы помечены явно маркером «per-cluster».

### Как использовать

По умолчанию YT-прагмы пишутся с префиксом `yt` — это означает, что настройка применится к текущему кластеру, на котором выполняется запрос. Чтобы та же прагма применилась к другому кластеру с другим значением, замените `yt` на имя этого кластера. Например, зададим в запросе разные временные директории для двух кластеров:

{% if audience == "internal" %}

```yql
PRAGMA yt.TmpFolder = "//tmp/my_folder";          -- текущий кластер
PRAGMA arnold.TmpFolder = "//tmp/other_folder";   -- кластер arnold

... -- тело запроса
```

В результате при выполнении запроса временные файлы будут сохраняться в `//tmp/my_folder` на текущем кластере и в `//tmp/other_folder` на кластере `arnold`.

{% else %}

```yql
PRAGMA yt.TmpFolder = "//tmp/my_folder";            -- текущий кластер
PRAGMA cluster_2.TmpFolder = "//tmp/other_folder";  -- кластер cluster_2

... -- тело запроса
```

В результате при выполнении запроса временные файлы будут сохраняться в `//tmp/my_folder` на текущем кластере и в `//tmp/other_folder` на кластере `cluster_2`.

{% endif %}

{% note warning %}

Нельзя одновременно использовать префикс `yt` и имя кластера.

Записи вида {% if audience == "internal" %} `PRAGMA arnold.yt.TmpFolder` {% else %} `PRAGMA cluster_2.yt.TmpFolder` {% endif %} или {% if audience == "internal" %} `PRAGMA yt.arnold.TmpFolder` {% else %} `PRAGMA yt.cluster_2.TmpFolder` {% endif %} недопустимы и приведут к ошибке.

{% endnote %}

Это пригодится для поиска и идентификации операций в архиве (поиск в нём доступен через [API](../../../api/python/userdoc.md#operation_and_job_info_commands)). ## yt.ColumnGroupMode {#column-group-mode}

| Возможные значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Строка: disable / single / perusage | disable | Статическая |

Задаёт режим вычисления поколоночных групп для промежуточных таблиц запроса. В режиме `disable` поколоночные группы не используются. В режиме `single` создается одна группа на все колонки таблицы. В режиме `perusage` создаются гранулярные группы колонок по их потребителям. Все колонки одной группы одновременно используются в одном или нескольких потребителях. Например, если в промежуточной таблице имеются колонки [a, b, c, d, e, f] и её используют две операции с выборками колонок [a, b, c, d] и [c, d, e, f] соответственно, то в таблице будут созданы три группы колонок [a, b], [c, d] и [e, f]. Если промежуточная таблица используется для публикации в выходную таблицу (т.е. потребителем является узел YtPublish), то поколоночные группы не применяются, за исключением явно заданного [модификатора column_groups](../insert_into.md#hints). В последнем случае, промежуточная таблица использует поколоночные группы модификатора.

## yt.ExternalTx {#external-tx}

Используется для указания внешней [транзакции](../../../user-guide/storage/transactions.md), в контексте которой будет выполняться запрос (например, если транзакция была создана через [CLI](../../../api/cli/commands.md#start-tx)). Это может понадобиться, чтобы выполнить несколько действий атомарно — например, 2 YQL-запроса или YQL-запрос и изменение таблиц. 

Также прагма позволяет прочитать промежуточные данные в транзакциях (например, в длительной операции, где есть временные файлы).

{% note info %}

Директории для работы запроса создаются в рамках указанной транзакции. Это может приводить к конфликтам при попытках записывать данные в ранее не существующую директорию с помощью двух запросов с разными ExternalTx.

{% endnote %}

| Возможные значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Строка с идентификатором транзакции | — | Статическая, [per-cluster](*per-cluster) |

#### Пример {#external-tx-example}

```yql
PRAGMA yt.ExternalTx = 'a3d149bd-674dfa1-fa68f314-2c22562';   -- ID транзакции (можно посмотреть через CLI)
```

#### Результат {#external-tx-result}

Запрос будет выполнен в рамках указанной транзакции.

#### Особенности и ограничения {#external-tx-limitations}

Указание прагмы без тела запроса вызовет ошибку.

## yt.InferSchema / yt.ForceInferSchema {#infer-schema}

| Возможные значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Число от 1 до 1000 | — | Статическая |

Вывод схемы данных по содержимому первых строк таблиц. Если PRAGMA указана без значения, то подразумевается, что по одной первой строке. Если указано несколько строк и типы данных для колонки встречаются разные, то они расширяются вплоть до Yson.

InferSchema включает вывод схемы данных только для тех таблиц, где она совсем не указана в метаданных. При ForceInferSchema схема данных из метаданных игнорируется, за исключением списка ключевых колонок для сортированных таблиц.

В дополнение к обнаруженным колонкам генерируется колонка-словарь _other (строка на строку) со значениями тех колонок, которые не присутствовали на первой строке, но нашлись где-то дальше. Это позволяет использовать на таких таблицах [WeakField](../../builtins/basic.md#weakfield).

Из-за широкого спектра возможных проблем, данный режим не рекомендуется к использованию и по умолчанию отключен.

## yt.JoinMergeUseSmallAsPrimary {#join-merge-use-small-as-primary}

| Возможные значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Флаг | - | Статическая |

Явное управление выбором primary таблицы в операции Reduce при Ordered JOIN стратегии. Если установлен в true, то в качестве primary таблички всегда будет выбираться меньшая сторона. Если значение флага равно false, то будет выбираться большая сторона за исключением случая уникальных ключей на большей стороне. Выбор большей таблицы в качестве primary безопасен даже при наличии в ней ключей-монстров, но работает медленнее. Если данная прагма не установлена, то выбор primary происходит автоматически исходя из максимального размера получающихся джобов (см. yt.JoinMergeReduceJobMaxSize).

После компрессии выходной размер может быть значительно меньше. В этом случае стоит увеличить значение прагмы на ожидаемый коэффициент компрессии.

## yt.OmitInaccessibleRows {#omit-inaccessible-rows}

Управляет поведением при чтении таблиц с [построчным ACL]({{yt-docs-root}}/user-guide/storage/row-level-security) (RLS).

По умолчанию чтение таблицы с установленным построчным ACL завершается ошибкой авторизации, если у пользователя нет права `full_read`. Прагма `yt.OmitInaccessibleRows` изменяет это поведение: при включении строки без доступа пропускаются, а запрос завершается успешно. В результат попадают только строки, разрешённые предикатом RLS.

| Возможные значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Флаг | false | Статическая |

#### Пример {#omit-inaccessible-rows-example}

```yql
PRAGMA yt.OmitInaccessibleRows = "true";

SELECT *
FROM `//path/to/table_with_rls`;
```

#### Результат {#omit-inaccessible-rows-result}

Запрос вернёт только строки, доступные текущему пользователю согласно построчному ACL. Строки, к которым нет доступа, будут пропущены без ошибки.

#### Ограничения {#omit-inaccessible-rows-restrictions}

- Нельзя указывать `row_index` в `ranges` при чтении таблицы с построчным ACL — запрос завершится ошибкой. Индексы строк в `ranges` отсчитываются относительно физических строк на диске, а не строк, доступных пользователю. Например, `//path/to/table[:#100]` вернёт до 100 строк с диска, часть из которых может быть недоступна и будет отфильтрована.
- RLS не поддерживается для динамических таблиц — любое чтение вернёт ошибку.

## yt.QueryCacheChunkLimit {#query-cache-chunk-limit}

Прагма используется для управления способом записи таблиц в кеш в зависимости от количества чанков в таблице: с помощью команды `concatenate` (как есть) или `merge` (с объединением).

| Возможные значения | По умолчанию | Вид |
| --- | --- | --- |
| Строка, которая содержит число чанков — `<Uint64>` | `'0'` | Динамическая |

#### Пример { #query-cache-chunk-limit-example }

```yql
PRAGMA yt.QueryCacheChunkLimit = '100000';
```

#### Результат { #query-cache-chunk-limit-result }

Если количество чанков в таблице меньше установленного лимита, то таблица записывается в кеш командой `concatenate` как есть.

Если количество чанков превышает заданный лимит — таблица записывается в кеш с помощью команды `merge` (с объединением чанков).

Таким образом, при значении по умолчанию `0` таблицы записываются в кеш через команду `merge`.

#### Особенности и ограничения { #query-cache-chunk-limit-constraint }

Команда `concatenate` — запись как есть — выполняется быстрее, чем `merge`, которой требуется время на объединение чанков и последующую запись. Таблицы с относительно небольшим количеством чанков выгоднее записывать в кеш через `concatenate`.

С помощью прагмы можно управлять сбросом кеша: включать или исключать номер ревизии при вычислении хеша из метаданных таблиц.

| Возможные значения | По умолчанию | Вид |
| --- | --- | --- |
| Строка, которая содержит `true` или `false` | `false` | Статическая |

#### Пример { #query-cache-ignore-table-revision-example }

#### Результат { #query-cache-ignore-table-revision-result }

## yt.QueryCacheMode {#query-cache-mode}

Прагма yt.QueryCacheMode используется для управления кешированием результатов MapReduce‑операций в {{product-name}}.

Кеш хранит результаты предыдущих операций: если такая же операция уже выполнялась в одном из предыдущих запросов, то она не будет запускаться. Вместо этого {{product-name}} возьмёт готовый результат из кеша. Это ускоряет работу запросов, которые содержат одинаковые с другими запросами вычисления.

Также Query Cache полезен в ситуации, когда вы отлаживаете или вносите относительно небольшие изменения в запрос: в этом случае результат выполнения значительной части операций берётся из кеша.

| Возможные значения | По умолчанию | Вид |
| --- | --- | --- |
| `'disable'` / `'readonly'` / `'refresh'` / `'normal'` | `'normal'` | Статическая |

#### Пример { #query-cache-mode-example }

```yql
PRAGMA yt.QueryCacheMode = 'disable';
```

#### Результат { #query-cache-mode-result }

В зависимости от выбранного режима {{product-name}} будет использовать, игнорировать или обновлять кеш:

- `disable` — кеш отключён. {{product-name}} не проверяет кеш и не записывает в него результаты. Режим подходит для продакшена, где данные часто меняются: он экономит ресурсы и исключает лишние операции чтения и записи в кеш;
- `readonly` — режим только для чтения. {{product-name}} берёт результат из кеша, если он есть, но не сохраняет в него новые результаты. Подходит для тестирования или отладки: можно использовать старые результаты и не засорять кеш новыми записями;
- `refresh` — режим только для записи. {{product-name}} сохраняет результаты в кеш, но не читает из него. Используется, чтобы обновить кеш: выполнить запрос, не читать из кеша и сохранить его результат для будущего использования;
- `normal` — режим по умолчанию. {{product-name}} и читает из кеша, и записывает в него. Особенно полезен при разработке и отладке, когда один и тот же запрос запускается многократно для ускорения итераций.

В режимах `normal` и `refresh` результат операций дополнительно сохраняется по пути `//<tmp_folder>/query_cache/<hash>`, где:

- `tmp_folder` — временная директория. По умолчанию — `tmp/<login>`, можно задать через прагму [yt.TmpFolder](#tmp-folder);
- `hash` — хеш от значимых метаданных и данных входных таблиц и логической программы, которая выполнялась в операции.

{% note info %}

В режиме `normal` при запуске MapReduce‑операции система ищет кеш по пути `//<tmp_folder>/query_cache/<hash>`. Если кеша нет, {{product-name}} проверяет, не запущена ли параллельно другая операция, которая вычисляет тот же кеш. Если такая операция есть, но ещё не записала результат, первая операция ожидает её завершения и затем берёт готовый кеш.

Учитывайте, что под операцию второго запроса, которая вычисляет кеш, может быть выделено меньше ресурсов, чем под операцию первого, который ожидает. Поэтому первый запрос будет выполняться медленнее, чем если бы он выполнялся без чтения из кеша.

{% endnote %}

## yt.RuntimeCluster {#runtime-cluster}

Задаёт текущее значение кластера для запуска MapReduce-операций. Подходит для выбора кластера, если в запросе есть таблицы, расположенные на разных кластерах.

Режим выбора кластера задаётся прагмой [`yt.RuntimeClusterSelection`](#runtime-cluster-selection).

| Возможные значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Строка c названием кластера | — | Динамическая |

#### Сигнатура {#runtime-cluster-signature}

```yql
PRAGMA yt.RuntimeCluster = '<cluster_name>';
```

{% note warning %}

Название кластера должно быть указано со строчной буквы. Например: `'{{ testing-cluster }}'`, а не `'{{ testing-cluster-name }}'`.

{% endnote %}

#### Пример {#runtime-cluster-example}

```yql
PRAGMA yt.RuntimeClusterSelection = 'force';
PRAGMA yt.RuntimeCluster = 'watt';
```

#### Результат {#runtime-cluster-result}

MapReduce-операции будут запущены строго на кластере Watt, так как выше задан режим `force` для прагмы `yt.RuntimeClusterSelection`.

#### Особенности и ограничения {#runtime-cluster-limitations}

Прагма `yt.RuntimeCluster` динамическая — можно использовать разные кластеры в рамках одного запроса, например:

```yql
PRAGMA yt.RuntimeClusterSelection = 'force';
PRAGMA yt.RuntimeCluster = "cluster1";

INSERT into cluster2.@tmp -- временная таблица на cluster2
SELECT ... ;  -- SELECT выполнится на cluster1

pragma yt.RuntimeCluster = "cluster2";

INSERT into cluster1.@tmp -- временная таблица на cluster1
SELECT ... ;  -- SELECT выполнится на cluster2

commit;

PRAGMA yt.RuntimeCluster = "cluster3";
SELECT ... from cluster1.@tmp as t1 JOIN cluster2.@tmp as t2; -- SELECT выполнится на cluster3
```

## yt.RuntimeClusterSelection {#runtime-cluster-selection}

| Возможные значения | По умолчанию | Статическая /<br/>динамическая |
| --- | --- | --- |
| Строка: disabe / auto / force | disable | Статическая |

Управляет режимом выбора кластера, на котором запускаются map-reduce операции. В режиме `disable` map-reduce операции запускаются, только если все входные таблицы операции (а также таблицы, использующиеся внутри лямбд операции) находятся на одном кластере.

В режиме `auto` операция запускается либо на кластере со входными таблицами (в случае, если все входные таблицы находятся на одном кластере), либо на кластере, который задан в текущем значении динамической прагмы.

В режиме `force` операция всегда запускается на кластере, заданном текущим значением динамической прагмы `yt.RuntimeCluster`.

Стоит отметить, что выходные таблицы операции всегда располагаются на том кластере, на котором она выполняется.

В режиме `auto`/`force` таблицы, приложенные явно или неявно (например, при выборе стратегии MapJoin) к операции, могут находиться на разных кластерах и никак не влияют на выбор кластера, на котором эта операция будет запущена. Кластер определяется только расположением входных таблиц и настройками `yt.RuntimeClusterSelection` и `yt.RuntimeCluster`.

## yt.StaticPool {#static-pool}

Используется для переопределения вычислительного пула, который по умолчанию является логином текущего пользователя.

Новое значение `yt.StaticPool` может быть только одно — если статическую прагму указать несколько раз, будет использоваться её последнее значение. Если нужно переопределить значение для следующего запроса, задайте его через динамическую прагму `yt.Pool`.

| Тип значения | Значение по умолчанию | Вид |
| --- | --- | --- |
| Строка | Логин текущего пользователя | Статическая, [per-cluster](*per-cluster) |

#### Сигнатура {#static-pool-signature}

```yql
PRAGMA yt.StaticPool = '<pool_1>';
```

#### Результат {#static-pool-result}

После указания прагмы запрос выполнится в пуле `<pool_1>`.

#### Особенности и ограничения {#static-pool-limitations}

- Указание несуществующего пула приведёт к ошибке выполнения запроса.

- При указании `yt.StaticPool` несколько раз используется последнее значение прагмы для всех запросов. Например, в `pool_2` будут выполнены и запросы, написанные после прагмы со значением `pool_1`, и запросы после прагмы с `pool_2`:

    ```yql
    PRAGMA yt.StaticPool = '<pool_1>';
    PRAGMA yt.StaticPool = '<pool_2>';
    ```
